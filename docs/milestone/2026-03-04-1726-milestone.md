## M1: Project Scaffolding + Foundation Modules

**Status:** `pending`
**Started:** --
**Completed:** --
**Dependencies:** --

### A. Description

Initialize the Tauri project (Rust backend + TypeScript frontend), implement the two foundational backend modules (Keychain Adapter, Preferences Store), set up the Tauri IPC whitelist structure, and configure design tokens CSS. This milestone produces the skeleton that all subsequent milestones build upon.

### B. Modules

| Module | Responsibility |
| --- | --- |
| Keychain Adapter | Implement macOS Security Framework integration -- store, retrieve, delete, list credentials |
| Preferences Store | Implement JSON file-based user preferences -- load, save, schema versioning, atomic writes |
| Tauri IPC | Scaffold command whitelist structure and AppError type |

### C. Requirements Covered

| ID | Description | Priority |
| --- | --- | --- |
| FR-PM-3 | Store API keys in macOS Keychain with encryption | Must |
| NFR-SEC-1 | API keys stored exclusively in macOS Keychain -- zero plaintext keys on disk | Must |
| NFR-SEC-7 | Tauri IPC whitelist -- only required commands exposed (scaffold) | Must |

### D. Acceptance Criteria

- [ ] Tauri project builds and launches on macOS 13+
- [ ] Keychain Adapter stores, retrieves, and deletes credentials via Security Framework
- [ ] Preferences Store reads/writes JSON with schema versioning and atomic file operations
- [ ] Tauri IPC scaffold with AppError type compiles
- [ ] Design tokens CSS (tokens.css) integrated into frontend
- [ ] Zero plaintext credentials on disk -- verified by test

### E. Notes

Preferences Store IPC commands (get_preferences, update_preferences) are deferred to M6 where the settings UI is built. The module itself is implemented here because Server Lifecycle (M4) depends on it to save last-used provider/region.

## M2: Provider Manager + Cloud Provider Integration

**Status:** `pending`
**Started:** --
**Completed:** --
**Dependencies:** M1

### A. Description

Define the CloudProvider trait and implement all three cloud provider SDKs (Hetzner, AWS, GCP). This covers API key validation, region listing with pricing, SSH key management, and server CRUD operations. Implement IPC commands for provider registration and region listing. Follow Risk R-7 mitigation: complete Hetzner first, then add AWS and GCP sequentially.

### B. Modules

| Module | Responsibility |
| --- | --- |
| Provider Manager | CloudProvider trait definition, provider registry, PricingCache with TTL |
| Hetzner Provider | hcloud-rs SDK integration -- validate, list regions, SSH keys, server CRUD |
| AWS Provider | aws-sdk-ec2 integration -- validate, list regions, key pairs, EC2 instances, security groups |
| GCP Provider | google-cloud-compute SDK integration -- validate, list regions, SSH metadata, instances, firewall rules |

### C. Requirements Covered

| ID | Description | Priority |
| --- | --- | --- |
| FR-PM-1 | Register API keys for Hetzner, AWS, and GCP | Must |
| FR-PM-2 | Validate API keys against the provider's API upon registration | Must |
| FR-RC-1 | Display available regions for each registered provider | Must |
| FR-RC-2 | Display the hourly cost per region retrieved from provider APIs | Must |
| FR-RC-4 | Select provider and region before server creation | Must |
| NFR-INT-1 | All 3 providers functional | Must |

### D. Acceptance Criteria

- [ ] CloudProvider trait defined with all 8 methods (validate_credential, list_regions, create_ssh_key, delete_ssh_key, create_server, destroy_server, get_server, list_regions)
- [ ] Hetzner provider passes integration test (validate key, list regions with pricing, server create/destroy)
- [ ] AWS provider passes integration test (same as above)
- [ ] GCP provider passes integration test (same as above)
- [ ] PricingCache implements ~1h TTL with stale fallback on API failure
- [ ] IPC commands register_provider, remove_provider, list_providers, list_regions functional
- [ ] Regions sorted by hourly cost ascending (FR-RC-3 is Should, but sorting is trivial)
- [ ] Invalid API key returns specific error per provider (AUTH_INVALID_KEY, AUTH_INSUFFICIENT_PERMISSIONS)

### E. Notes

Provider-specific cloud-init scripts are defined here (cross-cutting §6) but the provisioning orchestration is in M4. SSH key management (create_ssh_key, delete_ssh_key) is exposed via the trait here but orchestrated by Server Lifecycle in M4.

## M3: VPN Manager (WireGuard Integration)

**Status:** `pending`
**Started:** --
**Completed:** --
**Dependencies:** M1

### A. Description

Implement WireGuard tunnel management -- ephemeral key pair generation, config file lifecycle (write with permission 600, delete after use), wg-quick subprocess execution via sudo, and DNS leak prevention. This milestone is independent from Provider Manager (M2) and can be developed in parallel.

### B. Modules

| Module | Responsibility |
| --- | --- |
| VPN Manager | WireGuard key generation (Curve25519), config file management, wg-quick up/down subprocess, DNS routing configuration |

### C. Requirements Covered

| ID | Description | Priority |
| --- | --- | --- |
| FR-VC-1 | Generate WireGuard key pairs locally per session | Must |
| FR-VC-2 | Establish WireGuard tunnel to the provisioned server | Must |
| FR-VC-3 | Tear down WireGuard tunnel on user disconnect | Must |
| FR-VC-4 | Delete local WireGuard keys after tunnel teardown and server destruction | Must |
| FR-VC-5 | Route DNS queries through the VPN tunnel to prevent DNS leaks | Must |
| NFR-SEC-2 | Ephemeral WireGuard keys -- generated per session, deleted on teardown | Must |
| NFR-SEC-3 | DNS queries route through VPN tunnel during active session -- zero DNS leak | Must |
| NFR-SEC-6 | WireGuard config files have permission 600 and are deleted after use | Must |

### D. Acceptance Criteria

- [ ] WireGuard key pair generated using x25519_dalek or equivalent
- [ ] Config file written to /tmp with permission 600 -- verified by test
- [ ] Config file deleted immediately after wg-quick up -- regardless of success or failure
- [ ] wg-quick up/down executed via sudo osascript
- [ ] DNS server set to VPN server's resolver in WireGuard config
- [ ] Key pair zeroed from memory after tunnel teardown (zeroize crate)
- [ ] Bundle wireguard-go and wg-quick in Tauri app (ADR-0001)
- [ ] Integration test: tunnel up → DNS resolves through VPN → tunnel down → keys zeroed

### E. Notes

VPN Manager exposes a tunnel_up(server_ip, server_public_key) and tunnel_down() interface. It does not know about cloud providers -- Server Lifecycle (M4) orchestrates the connection between Provider Manager and VPN Manager. The wireguard-go and wg-quick binaries must be bundled in the Tauri sidecar per ADR-0001.

## M4: Server Lifecycle + Session Tracker

**Status:** `pending`
**Started:** --
**Completed:** --
**Dependencies:** M2, M3

### A. Description

Implement the central orchestrator (Server Lifecycle) that coordinates Provider Manager and VPN Manager for the full connect/disconnect flow. Implement Session Tracker for session state persistence and orphaned server detection. This is the largest milestone -- it wires together all backend modules into the complete server provisioning, connection, destruction, and recovery flows.

### B. Modules

| Module | Responsibility |
| --- | --- |
| Server Lifecycle | Orchestrate connect flow (SSH key → server provision → WireGuard tunnel), disconnect flow (tunnel down → destroy → verify deletion), auto-cleanup on failure |
| Session Tracker | Persist active session state (active-session.json), detect orphaned servers on app launch, report session status (IP, elapsed time, cost) |

### C. Requirements Covered

| ID | Description | Priority |
| --- | --- | --- |
| FR-SL-1 | Provision a cloud server with WireGuard via cloud-init on user request | Must |
| FR-SL-2 | Configure firewall rules during provisioning -- WireGuard UDP only | Must |
| FR-SL-3 | Destroy the server via provider API on user disconnect | Must |
| FR-SL-4 | Auto-cleanup a partially created server on provisioning failure | Must |
| FR-SL-6 | Detect orphaned servers from previous sessions on app launch | Must |
| FR-SL-7 | Prompt the user to destroy or reconnect to a detected orphaned server | Must |
| FR-SS-1 | Display the connected server's public IP during an active session | Must |
| FR-SS-2 | Display elapsed connection time during an active session | Must |
| NFR-SEC-5 | Provisioned server firewall allows only WireGuard UDP port | Must |
| NFR-PERF-1 | Server provisioning to WireGuard connection established ≤ 120 seconds | Must |
| NFR-PERF-2 | Disconnect to server destruction confirmed ≤ 30 seconds | Must |
| NFR-REL-1 | Detect orphaned servers on launch -- 100% detection across all providers | Must |
| NFR-REL-2 | Failed provisioning triggers auto-cleanup -- zero orphaned servers | Must |
| NFR-REL-4 | App crash does not leave WireGuard tunnel active without user awareness | Must |

### D. Acceptance Criteria

- [ ] Connect flow completes: SSH key gen → register → server provision → SSH key delete → WireGuard tunnel up → session file created → preferences updated
- [ ] Disconnect flow completes: tunnel down → server destroy → verify deletion → session file deleted → keys zeroed
- [ ] Auto-cleanup on any failure mid-connect: server destroyed, SSH key deleted, WireGuard keys zeroed
- [ ] Orphan detection on app launch: read session file → query provider API → return orphan list
- [ ] Resolve orphan: destroy (with verification) or reconnect (new WireGuard keys)
- [ ] Session status returns live-calculated elapsed time and accumulated cost
- [ ] Destruction verification: re-check server existence via get_server after destroy_server
- [ ] Retry destruction up to 3 times with exponential backoff on failure
- [ ] IPC commands functional: connect, disconnect, check_orphaned_servers, resolve_orphaned_server, get_session_status
- [ ] End-to-end test with real Hetzner API: connect → verify IP → disconnect → verify destruction

### E. Notes

This milestone's connect flow follows the 11-step sequence defined in API Design §4.C (connect command). The disconnect flow follows the verification pattern in cross-cutting concepts §8. Cloud-init scripts per provider are authored in M2 but orchestrated here. The SshKeyPair entity (data model §4.C) is managed entirely within Server Lifecycle -- generated before provisioning, provider-registered, then deleted after cloud-init completes.

## M5: Menu Bar UI -- Core Flow

**Status:** `pending`
**Started:** --
**Completed:** --
**Dependencies:** M4

### A. Description

Build the main Menu Bar UI with the core VPN flow: popover shell with stack navigation, disconnected view (provider selector, region list with cost, connect button), connected view (session card with IP/time/cost, disconnect button), provisioning stepper (3-step progress), destruction confirm dialog, error display patterns, and menu bar icon with 3 states. Integrate Liquid Glass design system, dark/light mode, and wire all views to backend IPC commands.

### B. Modules

| Module | Responsibility |
| --- | --- |
| Menu Bar UI | Popover structure, stack navigation, all core flow views, menu bar icon with status states, Liquid Glass styling |

### C. Requirements Covered

| ID | Description | Priority |
| --- | --- | --- |
| FR-MN-1 | Menu bar icon reflecting VPN status (disconnected, connecting, connected) | Must |
| NFR-SEC-7 | Tauri IPC whitelist -- all commands verified against allowlist (complete enforcement) | Must |

### D. Acceptance Criteria

- [ ] Popover opens on menu bar left-click with 320px fixed width
- [ ] Disconnected view: provider selector (if multiple), region list sorted by cost, connect button
- [ ] Connected view: session card (IP in SF Mono 22px, region + provider, elapsed time, cost), disconnect button
- [ ] Provisioning stepper: 3 steps (creating server → installing WireGuard → connecting tunnel) with done/active/waiting/failed states
- [ ] Destruction confirm dialog: modal over blur overlay, Cancel + Destroy buttons
- [ ] Error display: error glass card with retry option on provisioning failure
- [ ] Menu bar icon shows 3 states: disconnected, connecting (animated), connected
- [ ] Stack navigation with spring easing horizontal slide (650ms)
- [ ] Dark mode and light mode automatic via prefers-color-scheme
- [ ] Liquid Glass 4-layer sandwich applied to buttons and cards
- [ ] Keyboard navigation: Tab/Shift+Tab, Enter/Space, Esc to close
- [ ] All IPC commands match Tauri capabilities whitelist (fitness function S-3 pattern)
- [ ] Reduced motion: all animations → fadeIn 200ms when prefers-reduced-motion active

### E. Notes

The frontend polls get_session_status periodically to update the session card. The connect IPC command returns SessionStatus on success which populates the initial session card. Provider selector is visible only when multiple providers are registered (from list_providers). Last-used region pre-selected from preferences (expert shortcut -- UX Design §4.E). Orphaned server notification is displayed here after check_orphaned_servers returns results.

**Status:** `pending`

### D. Acceptance Criteria

- [ ] Onboarding detects first run (no providers registered via list_providers)
- [ ] Welcome screen → provider selection → API key input with stack navigation
- [ ] API key validation: loading state on button, input disabled during validation, inline error/success
- [ ] Provider-specific API key issuance summary with external doc link
