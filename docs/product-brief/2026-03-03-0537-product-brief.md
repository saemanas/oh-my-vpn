# Product Brief: Oh My VPN

## 1. Executive Summary

Oh My VPN is an open-source macOS menu bar application that automates on-demand VPN server provisioning across Hetzner, AWS, and GCP. Built with Tauri (TypeScript frontend + Rust backend) and powered by WireGuard, it lets users create, connect to, and destroy their own VPN server in one click. By replacing fixed-cost VPN subscriptions ($5--12/month) with pay-as-you-go cloud instances ($0.17--$4.63/month), Oh My VPN gives users full server control, zero-log guarantees, and dramatic cost savings.

## 2. Problem Statement

Commercial VPN services charge $5--12/month with opaque logging policies and no server control. Users must trust that providers do not log or sell their traffic -- a trust that has been repeatedly violated across the industry.

Self-hosting a VPN server solves the trust problem but introduces a high technical barrier: provisioning cloud instances, installing WireGuard, configuring firewall rules, managing keys, and manually tearing down servers to avoid unexpected charges. This process requires CLI fluency and cloud platform expertise, putting it out of reach for most users.

The result: privacy-conscious users either overpay for services they cannot trust, or give up on self-hosted VPN entirely.

## 3. Proposed Solution

A macOS menu bar application that abstracts the entire VPN server lifecycle into a simple GUI workflow:

1. **Register** -- Link cloud provider accounts via API key (encrypted in macOS Keychain)
2. **Select** -- Pick a provider and region with real-time hourly cost displayed
3. **Connect** -- One click to provision a server (cloud-init + WireGuard), generate keys, and establish the tunnel
4. **Disconnect & Destroy** -- One click to tear down the tunnel and delete the server

The app handles cloud API calls, WireGuard configuration, firewall rules, key generation/destruction, and error recovery automatically. Users interact only with the menu bar -- no terminal, no CLI, no manual configuration.

## 4. Target Users

### A. Privacy-Conscious Individuals

- Do not trust commercial VPN log policies
- Want full control over their VPN server
- Lack technical skills to self-host
- **Success criteria**: "I know my server has zero logs because I own it and destroy it after each session"

### B. Geo-Restriction Bypass Users

- Need access to region-locked content (Netflix, YouTube, etc.)
- Need access to country-restricted websites
- Use VPN on-demand, not 24/7
- **Success criteria**: "I can watch content from any supported region within 2 minutes"

### C. Cost-Sensitive Users

- Find $5--12/month VPN subscriptions wasteful for occasional use
- Prefer pay-as-you-go over fixed subscriptions
- **Success criteria**: "I pay less than $1/month for VPN"

### D. Developers and Power Users

- Already have cloud accounts and API keys
- Capable of CLI but tired of repetitive manual work
- Interested in advanced configuration options (v2)
- **Success criteria**: "What used to take 10 minutes of CLI commands now takes one click"

## 5. Competitive Landscape

| Solution | Approach | Gap |
| --- | --- | --- |
| Commercial VPN (NordVPN, ExpressVPN, etc.) | Subscription, shared servers | No server control, opaque logs, fixed cost |
| vpn-on-demand (GitHub) | Terraform + Ansible + GitHub Actions | Developer-only, no GUI, complex setup |
| ServerlessVPN | $49 paid, Docker-based, multi-cloud | Paid, requires Docker knowledge |
| Hetzner WireGuard App | `--image wireguard` + web UI | Hetzner-only, manual CLI required |
| sarg.org.ru method | Shell script + cloud-init | Developer-only, no automation |

**Common gap across all alternatives**: No solution offers a GUI-based, multi-cloud, one-click VPN experience accessible to non-developers.

## 6. Key Differentiators

1. **GUI One-Click Automation** -- Menu bar driven server lifecycle. All existing open-source alternatives are CLI/script-based
2. **Multi-Cloud** -- Hetzner, AWS, and GCP unified under one interface. Most projects support a single provider
3. **Real-Time Cost Visibility** -- Hourly cost per region displayed at selection time via provider APIs
4. **Mobile Pairing via QR Code** (v2) -- Connect iPhone instantly using official WireGuard app, no custom mobile app needed
5. **Open-Source + Pay-As-You-Go** -- App is free, users pay only cloud costs. Up to 97% cheaper than commercial VPN

## 8. Constraints and Risks

### A. Technical Constraints

| Constraint | Detail |
| --- | --- |
| macOS only | Tauri menu bar app, initial version targets macOS exclusively |
| WireGuard dependency | Requires WireGuard client on macOS (or built-in tunnel implementation) |
| Cloud CLI/API dependency | Requires hcloud, aws, gcloud CLI or HTTP API calls |
| cloud-init boot time | Latency between server creation and WireGuard readiness |

### B. Business Constraints

| Constraint | Detail |
| --- | --- |
| Open-source | No revenue model, side project |
| Solo development | All development, design, and testing by one person |
| User cloud account required | Entry barrier -- users must sign up for cloud provider and issue API key |

### C. Security Risks

| Risk | Impact | Mitigation |
| --- | --- | --- |
| API key leakage | Cloud account hijack, unauthorized server creation, billing explosion | macOS Keychain encrypted storage, minimize plaintext in memory |
| WireGuard key leakage | VPN traffic interception | Delete key pairs on server destruction, generate new keys per session |
| DNS leak | Real IP/location exposed during VPN session | Force DNS through VPN tunnel |
| IPv6 leak | Real IP exposed via IPv6 when only IPv4 is tunneled | Disable IPv6 or tunnel both protocols |
| Server-side log persistence | Access logs remain on server before destruction | Disable logging in cloud-init, server destruction deletes disk |
| MITM during provisioning | cloud-init script tampering | HTTPS API communication, generate WireGuard keys locally |
| WebView vulnerability (Tauri) | Frontend exploit to execute Rust backend commands | Strict Tauri CSP, IPC command whitelist |
| App update tampering | Malicious update distributed to all users | Code signing, HTTPS update server, checksum verification |
| SSH key exposure | Provisioning SSH key leaked allows server access | Ephemeral SSH keys per session, Keychain storage |
| Local config file theft | Malware reads config files | Sensitive data in Keychain only, never in config files |
| WireGuard config file exposure | `.conf` file contains private key | File permission 600, delete immediately after use |
| Cloud firewall misconfiguration | Provisioned server has open ports beyond WireGuard | cloud-init enforces firewall rules (WireGuard UDP port only, SSH blocked) |
| Server persists after app crash | Server remains running without user awareness, incurring charges | Detect active servers on app restart, notify user, crash recovery logic |

### D. Assumptions to Validate

1. cloud-init WireGuard auto-configuration works reliably across all 3 providers
2. Non-developers can complete cloud sign-up and API key issuance with onboarding guide alone
3. Tauri can control WireGuard tunnel without macOS permission issues
4. DNS/IPv6 leak prevention is achievable with WireGuard default config or requires additional firewall rules
5. Cloud providers fully delete disk data on server destruction
6. Tauri IPC security model adequately isolates sensitive cloud API calls
7. Each provider's firewall/security groups can be consistently configured at cloud-init stage
