# PRD: Oh My VPN

## 1. Introduction

### A. Product Overview

Oh My VPN is an open-source macOS menu bar application that automates on-demand VPN server provisioning across Hetzner, AWS, and GCP. Built with Tauri (TypeScript frontend + Rust backend) and powered by WireGuard, it lets users create, connect to, and destroy their own VPN server in one click. By replacing fixed-cost VPN subscriptions ($5--12/month) with pay-as-you-go cloud instances ($0.17--$4.63/month), Oh My VPN gives users full server control, zero-log guarantees, and dramatic cost savings.

### B. Document Scope

This PRD covers the full product lifecycle: MVP (v1.0), v1.1, and v2.0. All functional and non-functional requirements are defined with MoSCoW prioritization and release phasing.

## 2. User Stories

### A. Privacy-Conscious Individuals

- **US-PRI-1**: As a privacy-conscious user, I want to create my own VPN server so that no third party can log my traffic.
- **US-PRI-2**: As a privacy-conscious user, I want to destroy the server after each session so that no access logs persist.
- **US-PRI-3**: As a privacy-conscious user, I want my API keys stored in macOS Keychain so that credentials are not exposed in plaintext.
- **US-PRI-4**: As a privacy-conscious user, I want to see my connected IP so that I can verify my traffic is routed through the VPN.

### B. Geo-Restriction Bypass Users

- **US-GEO-1**: As a geo-bypass user, I want to select a specific region so that I can access content locked to that country.
- **US-GEO-2**: As a geo-bypass user, I want the server ready within 2 minutes so that I can start streaming quickly.
- **US-GEO-3**: As a geo-bypass user, I want to switch regions easily so that I can access content from different countries.
- **US-GEO-4**: As a geo-bypass user, I want to see available regions per provider so that I can pick the optimal location.

### C. Cost-Sensitive Users

- **US-CST-1**: As a cost-sensitive user, I want to see the hourly cost per region before creating a server so that I can make an informed choice.
- **US-CST-2**: As a cost-sensitive user, I want to destroy the server immediately after disconnecting so that I stop incurring charges.
- **US-CST-3**: As a cost-sensitive user, I want to see elapsed time and running cost during a session so that I can control spending.
- **US-CST-4**: As a cost-sensitive user, I want to see regions sorted by hourly cost so that I can quickly identify the cheapest option.

### D. Developers and Power Users

- **US-DEV-1**: As a developer, I want to register multiple cloud provider API keys so that I can use whichever provider I prefer.
- **US-DEV-2**: As a developer, I want the entire provisioning automated so that I save time on repetitive CLI work.
- **US-DEV-3**: As a developer, I want to see the server's public IP and connection details so that I can verify the setup.
- **US-DEV-4**: As a developer, I want macOS notifications on status changes so that I can multitask while the server provisions.

## 3. Key User Journeys

### A. First-Run Onboarding Journey

```
App launch (first time)
  -> Onboarding screen: select cloud provider
  -> Enter API key -> validate key via provider API
  -> Key stored in macOS Keychain
  -> Confirmation: "Provider connected"
  -> (Optional) Repeat for additional providers
  -> Onboarding complete -> Menu bar ready
```

**Edge cases**:

- Invalid API key -- clear error message, retry prompt
- Network failure during validation -- retry with timeout
- Insufficient cloud permissions -- specific error indicating required permissions

### B. Connect Journey (Core Flow)

```
Click menu bar icon
  -> Select provider (if multiple registered)
  -> Select region -> see hourly cost
  -> Click "Connect"
  -> Server provisioning starts (loading indicator)
  -> cloud-init installs WireGuard, configures firewall
  -> WireGuard keys generated locally, config applied
  -> Tunnel established
  -> Menu bar icon changes to "connected" state
  -> Notification: "VPN Connected"
  -> Session panel: connected IP, elapsed time, running cost
```

**Edge cases**:

- Provisioning timeout -- auto-cleanup + retry option
- cloud-init failure -- destroy server, show error, offer retry
- WireGuard handshake failure -- destroy server, show error
- Cloud API rate limit -- backoff + user notification

### C. Disconnect and Destroy Journey

```
Click menu bar icon (while connected)
  -> Click "Disconnect"
  -> WireGuard tunnel torn down
  -> Server destruction API call
  -> Server deleted (confirmed via API)
  -> WireGuard keys deleted locally
  -> Menu bar icon changes to "disconnected" state
  -> Notification: "Disconnected, server destroyed"
```

**Edge cases**:

- Destruction API failure -- retry with exponential backoff, notify user if persistent
- App crash while connected -- on next app launch, detect orphaned server, prompt user to destroy
- Network loss during destruction -- queue destruction, retry on reconnect

### D. Crash Recovery Journey

```
App restarts after crash/force-quit
  -> Check for active servers across all registered providers
  -> If orphaned server found:
    -> Notification: "Active server detected from previous session"
    -> Prompt: "Destroy now?" / "Reconnect?"
  -> User chooses action
  -> Resume normal state
```

## 5. Non-Functional Requirements

### A. Performance

| ID | Requirement | Target |
| --- | --- | --- |
| NFR-PERF-1 | Server provisioning to WireGuard connection established | ≤ 120 seconds |
| NFR-PERF-2 | Disconnect to server destruction confirmed | ≤ 30 seconds |
| NFR-PERF-3 | App launch to menu bar ready (no onboarding) | ≤ 3 seconds |
| NFR-PERF-4 | Region list with cost data loaded after provider selection | ≤ 5 seconds |

### B. Security

| ID | Requirement | Target |
| --- | --- | --- |
| NFR-SEC-1 | API keys must be stored exclusively in macOS Keychain | Zero plaintext keys on disk |
| NFR-SEC-2 | WireGuard keys must be ephemeral -- generated per session, deleted on teardown | Zero key reuse across sessions |
| NFR-SEC-3 | DNS queries must route through VPN tunnel during active session | Zero DNS leak (verifiable via leak test) |
| NFR-SEC-4 | IPv6 traffic must be disabled or tunneled during active session | Zero IPv6 leak |
| NFR-SEC-5 | Provisioned server firewall must allow only WireGuard UDP port | Zero additional open ports |
| NFR-SEC-6 | WireGuard config files must have file permission 600 and be deleted after use | Zero persistent config files post-session |
| NFR-SEC-7 | Tauri IPC must whitelist only required commands | Zero unrestricted IPC access |

### C. Reliability

| ID | Requirement | Target |
| --- | --- | --- |
| NFR-REL-1 | App must detect orphaned servers on launch and notify user | 100% detection rate across all registered providers |
| NFR-REL-2 | Failed provisioning must trigger auto-cleanup (server destruction) | Zero orphaned servers from failed provisioning |
| NFR-REL-3 | Server destruction retry on API failure | ≥ 3 retries with exponential backoff |
| NFR-REL-4 | App crash must not leave WireGuard tunnel active without user awareness | Tunnel state reconciled on next launch |

### D. Integration

| ID | Requirement | Target |
| --- | --- | --- |
| NFR-INT-1 | Must support Hetzner Cloud API, AWS EC2 API, and GCP Compute Engine API | All 3 providers functional |
| NFR-INT-2 | API key validation must return clear error for insufficient permissions | Specific error message per provider |
| NFR-INT-3 | Must handle cloud provider API rate limits gracefully | Backoff + user notification, zero silent failures |

## 7. Release Phasing

### A. MVP (v1.0)

All **Must-have** items. Core value delivery: one-click VPN create/connect/destroy.

- Provider registration + Keychain storage (Hetzner, AWS, GCP)
- Region selection + cost display
- Server provisioning + WireGuard connect/disconnect/destroy
- Connected server IP, elapsed time display
- Menu bar icon with status reflection
- First-run onboarding flow
- Orphaned server detection and recovery
- DNS leak prevention
- Auto-cleanup on provisioning failure

**Must FR**: 20 | **Must NFR**: 11 | **Total**: 31

### B. v1.1

All **Should-have** and **Could-have** items. Experience polish.

- Provider removal and status display (FR-PM-4, FR-PM-5)
- Region cost sorting, accumulated cost display (FR-RC-3, FR-SS-3)
- IPv6 leak prevention (FR-VC-6, NFR-SEC-4)
- Retry UX improvements (FR-SL-5)
- Notifications, onboarding guide enhancement (FR-MN-2, FR-OB-2, FR-OB-3)
- Dark mode and light mode (FR-MN-3)
- API rate limit handling, error message improvements (NFR-INT-2, NFR-INT-3)
- App launch and region load performance optimization (NFR-PERF-3, NFR-PERF-4)
- Destruction retry with backoff (NFR-REL-3)

### C. v2.0

Product Brief out-of-scope features.

- QR code mobile pairing (iPhone via official WireGuard app)
- Kill Switch / Split Tunneling
- Auto-shutdown timer / idle detection
- Cost cap / monthly report / session history
- CLI interface
- Multi-device support
- Auto-recommend cheapest provider/region
- `brew install` distribution

## 9. Risks

| # | Risk | Impact | Likelihood | Mitigation |
| --- | --- | --- | --- | --- |
| R-1 | cloud-init behavior differs across providers | Specific provider connection failure | Medium | Separate cloud-init scripts per provider, independent testing |
| R-2 | macOS permission policy changes block WireGuard tunnel control | Core functionality broken | Low | Investigate Tauri permission request mechanisms, prepare Network Extension API fallback |
| R-3 | Orphaned server continues billing after app crash | Unexpected user cost | Medium | Mandatory orphaned server detection on app restart (FR-SL-6/7), provider server list API query |
| R-4 | Insufficient API key permissions cause provisioning failure | Onboarding dropout | High | Validate required permissions at key registration (FR-PM-2), provide minimum permission guide per provider |
| R-5 | Provider API changes or deprecation | Provider functionality broken | Low | Abstract provider API layer, enable independent replacement |
| R-6 | DNS/IPv6 leak prevention failure | Core privacy value compromised | Medium | Automate leak testing in MVP, include leak verification during connection |
| R-7 | Solo development burden supporting 3 providers | MVP release delay | High | Complete Hetzner first, then add AWS and GCP sequentially |
| R-8 | Non-developer cloud sign-up/API key issuance barrier | Target user dropout | High | Step-by-step onboarding guide with screenshots, per-provider documentation |
